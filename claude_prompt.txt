## FIX 1: History tab shows "faulting" instead of app name

### Root Cause
`backend/app.py` around line 685, inside the `history()` route:
```python
m = re.search(r'faulting application name:\s*([^\s,:\n]+)', msg, re.IGNORECASE)
```
Windows OS prepends "Faulting application name: " to what simulate_crash.ps1 writes, so the
stored message reads: "Faulting application name: Faulting application name: excel"
The current pattern matches on the first occurrence and captures "Faulting" (the next word).

### Fix
Replace that single regex line (line ~685) in `backend/app.py` with this duplicate-prefix-tolerant version:
```python
m = re.search(
    r'faulting application name:\s*(?:faulting app(?:lication)? name:\s*)?([^\s,:\n]+)',
    msg, re.IGNORECASE
)
```

ALSO apply the exact same fix in `backend/event_log_monitor.py` in the `_extract_app_context()` function
(around line 270), where the same regex pattern is used:
```python
m = re.search(
    r'faulting application name:\s*(?:faulting app(?:lication)? name:\s*)?([^\s,:\n]+)',
    message, re.IGNORECASE
)
```

This fix must be applied in BOTH files. Do not change anything else in either function.

## FIX 2: Crashed app does not reopen after Approve

### Root Cause
The remediation script for Event 1000 crashes tries to restart the app by name. When `AppName` is
a bare name like "excel" or "word", `Start-Process excel` fails because PowerShell cannot resolve
the short name to an executable path. The script needs smarter process restart logic.

### Fix — Update `backend/app.py` in the inline PowerShell remediation block

Find the function `simulate_error1000_auto_fix` (around line 1099) or any inline powershell script
that handles Event 1000 app crash remediation. If the actual script is at a file path like
`Error1000_ApplicationCrash.ps1`, find and open that file.

In whichever file handles the "restart app" step, replace the naive `Start-Process $AppName` call
with this robust multi-path restart logic:
```powershell
# Attempt to relaunch the crashed application by trying common install locations
$appBase = $AppName -replace '\.exe$', ''
$launched = $false

# 1. Try direct process name (works if app is on PATH)
try {
    Start-Process "$appBase.exe" -ErrorAction Stop
    $launched = $true
    Write-Output "[OK] Relaunched $appBase via PATH"
} catch {}

# 2. Try common Office/application locations
if (-not $launched) {
    $searchPaths = @(
        "$env:ProgramFiles\Microsoft Office\root\Office16\$appBase.exe",
        "$env:ProgramFiles\Microsoft Office\Office16\$appBase.exe",
        "${env:ProgramFiles(x86)}\Microsoft Office\root\Office16\$appBase.exe",
        "${env:ProgramFiles(x86)}\Microsoft Office\Office16\$appBase.exe",
        "$env:LOCALAPPDATA\Microsoft\WindowsApps\$appBase.exe"
    )
    foreach ($path in $searchPaths) {
        if (Test-Path $path) {
            try {
                Start-Process $path -ErrorAction Stop
                $launched = $true
                Write-Output "[OK] Relaunched $appBase from: $path"
                break
            } catch {}
        }
    }
}

# 3. Fallback: try shell verb (opens via file association/Windows Store)
if (-not $launched) {
    try {
        Start-Process "shell:appsFolder\$appBase" -ErrorAction Stop
        $launched = $true
        Write-Output "[OK] Relaunched $appBase via shell:appsFolder"
    } catch {}
}

if (-not $launched) {
    Write-Output "[WARN] Could not automatically relaunch $appBase. Please reopen it manually."
}
```
