## What Was Fixed

### Critical Issue: Event 1000 Not Being Ingested
**Problem**: Windows Event Log contained real Event 1000 entries (application crashes), but the backend PowerShell query was not detecting them.

**Root Cause**: PowerShell `Get-WinEvent -FilterHashtable` with `StartTime` parameter returns 0 events. This is a known Windows PowerShell 5.1 quirk.

**Solution**: Modified `backend/event_log_monitor.py` lines 136-158 to:
1. Remove `StartTime` from FilterHashtable
2. Fetch up to 5000 events without date filtering  
3. Filter by TimeCreated in PowerShell pipeline (client-side)
4. Select first N events after filtering

This workaround bypasses the PowerShell bug while maintaining the same functionality.

## Database Schema

### events table
- `id`: Event row ID (auto-increment)
- `event_id`: Windows Event ID (e.g., 1000)
- `timestamp`: When event occurred (UTC)
- `source`: Event provider (e.g., "Application Error")
- `message`: Event message text (up to 2000 chars)
- `category`: Event category (from rule definitions)

### remediation_history table  
- `id`: Remediation attempt ID
- `event_row_id`: FK to events table
- `rule_id`: FK to rules table (which script to run)
- `status`: 'success', 'failed', 'error'
- `output`: Script stdout + stderr
- `created_at`: When remediation was attempted (UTC)

### rules table
- `id`: Rule ID
- `event_id`: Which Windows Event ID triggers this rule
- `auto_remediate`: 1 = auto-execute, 0 = manual approval required
- `name`: Human-readable rule name
- `description`: What the rule fixes
- `script_path`: Path to PowerShell script
